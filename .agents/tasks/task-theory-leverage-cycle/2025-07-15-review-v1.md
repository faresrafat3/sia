# Theory leverage cycle for Virtual-SIA pipeline

The change adds a "Local Theory Leverage" mechanism (B1-B5) that lets local theories influence three pipeline decisions: verification strictness, tier routing, and concept activation. The mechanism is gated behind `use_theory_leverage: bool = False`, preserving default behavior. Theories generate predictions about task difficulty/failure based on token overlap between their `predictive_claims` and task text, then the pipeline uses those predictions to tighten verification, prevent cheap tiers, and boost theory-aligned concepts. A Laplace-smoothed tracking of prediction accuracy feeds back into theory confidence over time.

**Watch for:** (1) The prediction-correctness feedback treats both `predicts_failure` and `predicts_difficulty` as expecting bad outcomes, which conflates severity levels and biases the theory toward always predicting failure (**confirmed**). (2) The concept-engine theory-guided admission re-iterates the `concepts` argument after it has already been consumed by `select_applicable_concepts`, which silently yields nothing when callers pass single-use iterators (**confirmed**). (3) The `secondary_hit` variable in `verify_output_theory_guided` is unconditionally reassigned after the `if/else` block, overwriting the difficulty-mode value with dead code that misleads readers (**confirmed**).

## High-level view

The gate works: when `use_theory_leverage=False`, no new code paths execute, and the return dict adds three keys (`use_theory_leverage`, `theory_prediction`, `theory_predictive_value`) unconditionally so downstream consumers always see them.

The prediction engine is a token-overlap heuristic with a 1-token match threshold. The tokenizer doesn't filter stop words, so claims containing "tasks", "with", "will" match nearly every in-scope task text spuriously. This means prediction confidence will be high on most tasks regardless of actual claim relevance.

The predictive-value feedback loop uses identical logic for `predicts_failure` and `predicts_difficulty`: both treat a bad outcome as "prediction correct." The system cannot learn that a theory overpredicts difficulty but correctly predicts failure — both signals feed the same counter, creating a monotonic incentive to over-predict.

The concept-engine theory-guided admission stores concept IDs in a `List[ConceptCard]`-annotated variable (actually strings) then resolves them by re-iterating the input `concepts`. The test suite always passes lists, masking the iterator-consumption bug.

The pipeline's verification+theory+anomaly logic is duplicated verbatim in the pre-escalation and post-escalation blocks (~25 lines each). They are identical today, but any future edit must be applied to both or they diverge silently.

<summary>Issues (5)</summary>

1. **Prediction correctness conflation** — `predicts_failure` and `predicts_difficulty` both count a bad outcome as "correct." Separate counters (or at minimum differentiate the conditions) so the Laplace-smoothed value reflects actual predictive accuracy per prediction type.
2. **Iterator-consumption in concept admission** — `select_applicable_concepts_theory_guided` passes `concepts` to `select_applicable_concepts` (consuming the iterable), then later calls `list(concepts)` to resolve admitted IDs. If a generator is passed, the second iteration yields nothing and admission silently fails. Materialize the iterable at function entry.
3. **Stop-word pollution in prediction matching** — `_tokenize` doesn't filter stop words. Claims like "Tasks with X will fail" match any task text containing "tasks" or "with," producing spurious predictions. Consider a minimum token length or a stop-word set.
4. **Shadowed `secondary_hit` in verification** — In `verify_output_theory_guided`, the difficulty-mode branch computes `secondary_hit` for use in `strict_evidence_ok`, but then line 288 unconditionally reassigns `secondary_hit` to a fresh computation. Correctness is preserved but the reassignment is dead code.
5. **`admitted` type mismatch** — `admitted: List[ConceptCard]` is declared but populated with `concept_ref` strings. The resolution loop works because both sides are strings, but the type annotation is wrong.

## Prediction-correctness feedback conflates severity levels

In the pipeline post-processing block:

```python
if theory_prediction["predicts_failure"]:
    prediction_correct = not actual_good_enough
elif theory_prediction["predicts_difficulty"]:
    prediction_correct = not actual_good_enough
else:
    prediction_correct = actual_good_enough
```

Both branches resolve identically. A theory that predicts difficulty (a softer claim) shouldn't need outright failure to be validated — difficulty might manifest as needing escalation, taking longer, or barely passing. A theory that predicts failure should only be rewarded when the task actually fails hard. The current logic means a theory can improve its `predictive_value` by always predicting difficulty on hard tasks. This creates a monotonic incentive to over-predict because hard tasks frequently fail verification anyway.

## Iterator consumption in concept-engine theory-guided admission

`select_applicable_concepts_theory_guided` accepts `concepts: Iterable[ConceptCard]` and delegates to `select_applicable_concepts(... concepts ...)`, which iterates the entire collection. Later, if admission is needed:

```python
all_concepts_list = list(concepts) if not isinstance(concepts, list) else concepts
for concept in all_concepts_list:
    if concept.id in admitted and concept not in selected:
        selected.append(concept)
```

If `concepts` was a generator or any single-pass iterable, `list(concepts)` yields `[]` because the elements were already consumed. The `isinstance(concepts, list)` check catches the common case, but any caller using a generator, `filter()`, or chained itertools will hit this silently. Fix: `concepts_list = list(concepts)` at function entry, pass `concepts_list` to both call sites.

## Token overlap without stop-word filtering

`_tokenize` splits on whitespace and strips punctuation. The claim "Tasks with insufficient contrast will fail" tokenizes to `{"tasks", "with", "insufficient", "contrast", "will", "fail"}`. A task text "Compare tasks with evidence" matches on "tasks" and "with" — two common English words — producing a spurious relevant-claim match. The threshold for `check_theory_explains_contradiction` is 2 tokens and for `get_theory_prediction_for_task` is just 1 token, so stop words dominate matching. The practical effect: prediction confidence will be the theory's full `predictive_value` on nearly every in-scope task, reducing the signal's discrimination.

## Verification duplication

The pipeline contains two near-identical verification blocks (pre-escalation and post-escalation), each ~25 lines following the same `if theory_leverage → theory-guided → layer anomaly on top; else anomaly-aware` pattern. A helper function extracting this logic would eliminate the risk of future divergence.

## Test coverage gaps

**Not tested:** (a) Iterator-consumption scenario — all tests pass lists. (b) Pipeline behavior when theory prediction has no `relevant_claims` (confidence = 0.0, hitting neither routing threshold). (c) `explanatory_power` accumulates but has no downstream consumer in this cycle — no test verifies it influences any decision.
