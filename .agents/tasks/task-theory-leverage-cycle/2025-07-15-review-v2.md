# Theory leverage cycle for Virtual-GENESIS pipeline (v2)

The change adds a "Local Theory Leverage" mechanism (B1-B5) that lets local theories influence verification strictness, tier routing, and concept activation. Theories generate predictions via token overlap between their `predictive_claims` and task text, then the pipeline uses those predictions to tighten verification, prevent cheap tiers, and boost theory-aligned concepts. A Laplace-smoothed tracker of prediction accuracy feeds back into theory confidence. The mechanism is gated behind `use_theory_leverage: bool = False`, preserving default behavior. All five v1 issues have been addressed.

**Watch for:** (1) The prediction-correctness conflation remains by design (documented as a known simplification), but the documentation doesn't capture the specific failure mode: a theory that always predicts difficulty gets rewarded whenever verification fails, creating a monotonic drift toward over-prediction (**confirmed**, accepted per implementation memo). (2) Verification duplication between pre-escalation and post-escalation blocks (~25 lines each) remains identical with no extraction into a helper, making future divergence likely (**confirmed**).

**Verdict**: APPROVED

## High-level view

All five v1 issues were resolved. The iterator-consumption bug is fixed by materializing `concepts` to a list at function entry. The type annotation was corrected to `List[str]`. The test suite expanded from 88 to 126 tests (38 new) covering all six new functions and pipeline integration with both leverage flags.

## Verification duplication risk

The pipeline contains two near-identical verification blocks:

```python
# Pre-escalation (lines ~115-130):
if use_theory_leverage and theory_prediction and (...):
    verification = verify_output_theory_guided(...)
    if use_anomaly_leverage and anomaly_severity > 0.5:
        anomaly_verification = verify_output_anomaly_aware(...)
        if not anomaly_verification[...]["good_enough"]:
            verification = anomaly_verification
else:
    verification = verify_output_anomaly_aware(...)

# Post-escalation (lines ~155-170): identical structure
```

If one block gains a new condition (e.g., a third leverage type, or a changed severity threshold), the other must be updated manually. A `_run_verification(...)` helper would eliminate this risk.

## Test coverage gaps

**Not tested:** (a) The exact escalation path where theory-guided verification triggers post-escalation — tests exercise the pipeline but don't force escalation with theory leverage active. (b) Pipeline behavior when `theory_prediction` has `relevant_claims` but neither `predicts_failure` nor `predicts_difficulty` is True (neutral prediction with non-zero confidence).
