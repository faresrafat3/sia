# Broader domain expansion: analysis, extraction, planning families

Three new task families (analysis, extraction, planning) with 18 task cases, a domain transfer measurement module, a broader domain eval runner, and 37 new tests. The goal is to answer whether the cognitive mechanisms built for comparison/synthesis/procedure generalize to structurally different task families. The evaluation infrastructure correctly reports a 0.0 transfer rate in the synthetic pipeline (no real concept formation occurs in simulation), which is expected. Classification accuracy on the new families is 83% — but the 17% miss rate is entirely extraction cases being swallowed by the pre-existing `procedure` family due to keyword collision.

**Watch for:** Keyword collision between `procedure` and `extraction` families causes 3/6 extraction tasks to misclassify (confirmed). The `"plan"` keyword in the planning family matches inside unrelated words like "transplant" (confirmed, low-severity in practice). The domain transfer report always returns 0.0 transfer_rate in the simulated pipeline, meaning the portability verdict is structurally untestable without a live model (likely architectural gap).

---

## High-level view

The classification system uses substring-in-string matching against keyword lists, and the pre-existing `procedure` family's keywords (`extract`, `format`, `parse`, `fields`, `structured`) overlap heavily with the new `extraction` family's intended domain. Three of six extraction task prompts lose the keyword race to `procedure` because they contain more procedure-matching substrings than extraction-matching ones. This is a semantic boundary problem introduced by this diff, not a pre-existing issue — the extraction family didn't exist before, so the overlap didn't matter.

The domain transfer measurement infrastructure is well-designed as a framework but structurally inert in the current eval pipeline. Since `compare_conditions` runs a simulated pipeline that never produces real concept activations (no blackboard entries have `concept_activations` populated), the transfer report will always compute 0.0. The code is correct as a measurement tool, but the eval runner can't exercise it meaningfully until concept formation actually runs — making the "portability verdict" a dead letter in any automated test run.

The concept engine configuration for new families is conservative and coherent with the existing pattern. Analysis and planning get `semantic_balanced` with `max_active: 2`; extraction gets `contract_heavy` with `max_active: 1` and a high `min_score: 7`. This mirrors the comparison/synthesis/procedure pattern and reflects the intuition that extraction tasks need precision over breadth.

The latter case (concepts activated on new families but none on old families) semantically isn't "perfect transfer" — it's "no baseline to compare against." (confirmed)

## Procedure/extraction keyword collision

The `procedure` family keyword list contains: `extract`, `format`, `parse`, `fields`, `attributes`, `structured`. The new `extraction` family keyword list uses multi-word phrases like `"extract structured"`, `"parse fields"`, `"extract all"` — but the classifier uses `if k in text` matching, which means the single-word `"extract"` from procedure triggers on every text that also contains `"extract structured"`.

Running the actual classification on the 6 extraction task prompts:

```
Extract structured incident data... → procedure (scores: procedure=4, extraction=3)
Parse fields from the mixed-format... → procedure (scores: procedure=5, extraction=3)
Tabulate the key information from... → procedure (scores: procedure=4, extraction=3)
```

The extraction cases that succeed do so because they lean harder on extraction-only keywords (`"pull out"`, `"identify entities"`, `"data points"`, `"key information"`) that don't collide with procedure. The 50% misclassification rate means the extraction family is only half-functional as a distinct classification target.

The test suite tests classification with an idealized prompt (`"extract structured data points from the noisy text and tabulate the key information"`) that packs enough extraction-specific keywords to win the race, but this doesn't reflect the actual task set's prompts.

## Domain transfer metric inertness

`generate_domain_transfer_report` iterates over `task_results`, looking for `run.get("blackboard", {}).get("concept_activations", [])`. The `compare_conditions` function (used by the eval runner) produces task_results where each run's blackboard contains `concept_activations: []` because the simulated pipeline doesn't run real concept formation.

The code is structurally sound — the transfer coefficient calculation, the family-pair matrix, the per-concept breakdown are all logically correct. The unit tests in `TestDomainTransferReport` verify this with manually-constructed mock results that DO contain concept activations. So the function works when fed real data. The gap is at the integration level: the eval runner will never produce non-zero transfer metrics.

This creates a disconnect between the documentation (which describes a "portability verdict" as a key deliverable) and what the system can actually produce. The Broader Domain Cycle Memo describes detailed decision logic ("if overall_portable = True: the mechanisms generalize") but this verdict will always be "not portable" until a live model integration exists.

## Substring matching fragility

The classifier uses `sum(1 for k in keys if k in text)` — pure substring containment. The keyword `"plan"` will match inside "transplant", "biplane", "misexplanation". In practice, the task prompts in this PR are carefully crafted to avoid this, and the classifier's tie-breaking (highest score wins) usually rescues correct classification. But it means the classification system's robustness degrades for arbitrary user inputs.

This isn't a new design flaw (all existing families use the same mechanism), but the planning family's `"plan"` keyword is shorter and more common as a substring than any existing keyword. The analysis family's `"analyze"` and the extraction family's multi-word phrases don't suffer from this issue.

## Test coverage shape

The 37 new tests cover:
- Enum existence (2 tests)
- Keyword presence and classification correctness (9 tests, including Arabic keywords)
- Success criteria mapping (3 tests)
- Concept engine config values (6 tests)
- Task set structure validation (9 tests)
- Domain transfer report with mock data (5 tests)
- Eval runner integration (2 tests, one executes the full pipeline)

Not tested: the 3/6 extraction misclassification is not surfaced as a failure because the classification test uses a prompt specifically designed to succeed rather than testing the actual task set prompts. An assertion like `for case in EXTRACTION_CASES: assert classify(case.prompt_text.lower())[0] == "extraction"` would expose the collision immediately but doesn't exist.
