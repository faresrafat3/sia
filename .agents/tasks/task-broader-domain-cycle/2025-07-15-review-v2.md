# Broader domain expansion: analysis, extraction, planning families (v2)

Three new task families (analysis, extraction, planning) with 18 task cases, a domain transfer measurement module, a broader domain eval runner, and 37 new tests. All four issues from the v1 review are resolved: the procedure/extraction keyword collision is fixed via disambiguation logic and keyword removal, the "plan" substring false-positive is fixed with a leading-space approach, the transfer rate division semantics now return 0.0 when no baseline exists, and the domain transfer report's structural inertness is documented as a known limitation. Classification accuracy on the new families is now 18/18 (100%).

**Watch for:** The disambiguation logic only fires when procedure leads extraction by at most 1 point — a text where procedure leads by 2+ points with extraction signals present will still misclassify (possible, not triggered by any current task prompt but could affect future prompts). The `second_score` variable is computed twice in `classify_task_family`, once before and once after disambiguation — the first assignment is dead code (confirmed, cosmetic).

**Verdict**: APPROVED

---

## High-level view

The keyword collision fix works through two complementary changes: removing `"extract"` from the procedure keyword list (so procedure no longer free-rides on text containing "extract") and adding a disambiguation block that checks for multi-word extraction signals when procedure wins by at most 1 point. The combination achieves 100% classification on the current 18-case task set, with the extraction prompts specifically enriched with enough extraction-specific vocabulary to win the keyword race even without disambiguation.

The leading-space approach to `" plan"` eliminates false positives from words like "transplant" and "biplane" — verified that these produce zero planning score. The trade-off is that `"plan"` at the very start of a string (position 0) won't match, but task prompts are sentences that naturally contain a leading space before "plan".

The domain transfer report now returns 0.0 when `total_old == 0` regardless of `total_new`, which is the correct semantic choice (no baseline means no measurable transfer). The report's structural inertness in the simulated pipeline is acknowledged as a known limitation in the codebase rather than treated as a bug.

## Disambiguation logic design

The disambiguation block at lines 107-116 of `service.py` activates under three conditions: (1) procedure won, (2) extraction is in the ranked list, and (3) the score margin is at most 1. When all three hold, it counts how many `_EXTRACTION_PRIORITY_SIGNALS` (multi-word phrases like "extract structured", "key information") appear in the text. If 2+ signals match, it promotes extraction to first place in the ranked list.

The margin threshold of 1 means the disambiguation is conservative — it won't override a strong procedure classification. For the current task set this is sufficient because the prompts were rewritten to contain enough extraction-specific vocabulary that extraction usually wins outright (scores of 4-7 for extraction vs 1-4 for procedure). The disambiguation serves as a safety net for edge cases rather than the primary classification mechanism.

One behavioral quirk: after a disambiguation flip, `best_score` is set to the extraction score (which is <= the procedure score by definition, since procedure won). This means the ambiguity check (`second_score >= max(1, best_score - 1)`) will always be True post-flip. This is semantically correct — the families genuinely are close when disambiguation fires — so it signals downstream consumers that the classification has low confidence.

## Test coverage adequacy

The test suite verifies classification with a single representative prompt per family (`test_classifies_extraction_text` etc.) but does not run the classifier against all 18 actual task prompts. Running this manually confirms 18/18 pass, but a regression test that iterates over `PROTOTYPE_V7_BROADER_DOMAIN_CASES` and asserts `classify_task_family(case.prompt_text.lower())[0] == case.expected_primary_family` would catch future keyword drift. This is a suggestion, not a blocking concern — the eval runner's `classification_report` effectively performs this check at runtime and reports the match rate.
