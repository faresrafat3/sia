# Identity governance and paradigm forking for Virtual-SIA

The change adds two gated capabilities to the cognitive agent pipeline: identity governance (drift detection against declared commitments, with advisory recommendations) and paradigm forking (crisis detection triggering self-redesign under safety constraints). Both features default to off and don't alter existing behavior. Eight new Arabic "Legitimate Theft" entries and two implementation memos document the theoretical grounding.

**Watch for:** Governance operates on the *output* after the pipeline has already committed to a reasoning path — it cannot prevent a drifted decision, only log it after the fact (confirmed). The `identity.drift_score` used for crisis detection is a stale value passed in by the caller, never updated dynamically within the pipeline (confirmed). Documentation states governance thresholds of 0.3/0.6 but the code implements 0.5/0.7 (confirmed).

## High-level view

Identity governance acts as a post-hoc advisory check: after reasoning and verification are complete, the pipeline computes token overlap between the final claim text and the identity's commitments, then emits a recommendation. Because it runs after the output is locked in, it can log concern but cannot gate the output. The `CommitmentLedger` class is built and tested but never wired into the pipeline — it's infrastructure waiting for a future integration.

Crisis detection feeds from the memory store's failure history and the theory registry's low-performing theories. When all three thresholds are met simultaneously (anomaly count, theory failures, drift), the paradigm fork protocol generates a proposal that splits commitments into preserved/discarded halves and executes under three safety guards. The fork's `current_cycle` proxy uses `len(store.all())`, which is stable within a single run but will drift from true invocation count if forgetting is active.

The drift metric — token overlap between commitment text and decision text — is a lexical proxy. Semantically equivalent decisions expressed in different vocabulary register as high drift. Negation is invisible to the metric: "do not ensure transparency" scores identically to "ensure transparency."

## Governance as advisory rather than gating

The pipeline ordering is: reasoning → verification → escalation → anomaly extraction → paradigm fork → theory update → **identity governance** → store episode memory. By the time governance runs, the blackboard's `candidate_claims[0]` is the final output. The governance check computes drift against that output and, if not aligned, appends to `identity.accountability_log` — but does not modify the output, prevent storage, or trigger re-reasoning.

The pipeline can produce and store a drifted output where the only signal is a dict entry in the return value. If a caller doesn't inspect `alignment_report`, the violation passes silently. The memo says "advisory not blocking," but this weakens the H9 "governance as precondition" claim from the theory doc (section 17).

## Drift score lifecycle gap

`detect_crisis` on line 279 of `minimal_run.py` consumes `identity.drift_score` directly:

```python
crisis_report = detect_crisis(anomaly_history, theory_failures, identity.drift_score)
```

This value is whatever the caller set on the identity object before invoking the pipeline. The pipeline never calls `measure_drift` to update this field before crisis detection — even though the governance step (which does compute live drift) runs *after* the fork. A caller must manually maintain `drift_score` across pipeline invocations for crisis detection to trigger correctly.

If both features are enabled simultaneously, the ordering creates an asymmetry: the fork uses stale drift, but governance uses freshly computed drift. A fork that resets drift to 0.0 will then see governance compute a potentially-high drift against the new identity's (altered) commitments.

## Token overlap as drift metric

The drift detector tokenizes by whitespace-splitting and lowercasing, then computes `1 - |commitment_tokens ∩ behavior_tokens| / |commitment_tokens|`. Two failure modes:

1. **Negation blindness.** A decision can score zero drift by repeating all commitment words verbatim regardless of meaning. "I will not ensure transparency in reasoning" shares all tokens with "ensure transparency in reasoning" and reads as perfectly aligned.

2. **Asymmetric normalization.** The metric normalizes by commitment token count only. An extremely long decision that happens to include all commitment words shows zero drift even if 95% of its content is unrelated.

## Paradigm fork safety constraints

The justification guard is effectively dead for internal proposals — `propose_fork` always generates justification from crisis reasons, so this check only fires for externally-constructed proposals with empty strings.

The preserved/discarded overlap guard checks via set intersection. Since `propose_fork` places lineage items in preserved and second-half commitments in discarded, overlap only occurs if a commitment string literally equals a lineage string. This isn't guarded at proposal time, so a commitment like "v0.1" (matching a lineage entry) would silently fail the fork.

The `propose_fork` function mechanically splits commitments in half (first half preserved, second half discarded) without assessing which commitments actually failed — placeholder scaffolding for the research prototype.

## Pipeline return value for fork consumers

The pipeline reassigns the local `identity` variable after a successful fork (for downstream governance), but doesn't return the identity object in the result dict. The only way for a caller to obtain the new identity is `result["fork_result"]["new_identity"]`. The memo doesn't document this consumption pattern.

## Test coverage gaps

Not tested:
- Both `use_identity_governance` and `use_paradigm_fork` enabled simultaneously (the interaction path where fork produces a new identity that governance then checks).
- Edge case where `propose_fork` generates a preserved/discarded overlap (commitment text equals a lineage entry).
- Behavior when `blackboard.candidate_claims` is empty (the governance check unconditionally indexes `[0]`).
