# Family-specific concept selectivity, test scaffold, and deep analysis documentation

This change enhances the concept engine's activation scoring to use per-family strategies (`contract_heavy`, `semantic_balanced`, `structural_only`), fixes a config variable name bug in the selectivity ablation runner, replaces the retriever's global concept cap with a per-family cap, adds a `pyproject.toml` for packaging, introduces 11 pytest tests, and creates three documentation files that describe the concept engine architecture, eval system results, and frozen regime status.

Watch for: The `semantic_balanced` secondary admission rule is dead code under current config (confirmed). The `structural_only` and `contract_heavy` scoring paths are either unreachable or produce the same result as the default (confirmed). The retriever's per-family cap can silently drop concepts selected from secondary candidate families (likely).

## High-level view

The strategy-differentiated scoring in `apply.py` introduces three code paths, but the current configuration renders two of them inert. The `contract_heavy` strategy doubles `contract_fit` in the score, which only matters when a `task_contract` is provided — and the tests never supply one, leaving this path unvalidated. The `structural_only` path computes the same formula as the default. The `semantic_balanced` secondary admission rule requires `fam_limit >= 2`, but synthesis has `max_active=1`, so the rule can never fire.

The retriever change replaces the global cap (`DEFAULT_GLOBAL_MAX_ACTIVE_CONCEPTS`) with a per-family cap derived from the primary task family. This introduces an asymmetry: concepts selected by secondary candidate families during the multi-family loop may be silently truncated to the primary family's limit — potentially zero for procedure tasks even if a secondary family contributed valid concepts.

The test suite does not exercise the strategy-differentiated scoring or the secondary admission rules. The `contract_heavy` 2x multiplier, the `structural_only` high-threshold fallback, and the retriever per-family cap all lack test coverage.

## Strategy-differentiated scoring: two inert branches

The core change in `apply.py` adds a strategy switch at scoring time:

```python
if strategy == 'contract_heavy':
    base_score = family_fit + (contract_fit * 2) + semantic_fit
elif strategy == 'structural_only':
    base_score = family_fit + contract_fit + semantic_fit
else:
    base_score = family_fit + contract_fit + semantic_fit
```

The `structural_only` branch produces the identical formula as the `else` (semantic_balanced) branch. The only behavioral difference for procedure tasks is the high-threshold fallback further down (`adjusted >= 10`), not the scoring formula itself. The scoring switch for `structural_only` gives the appearance of differentiated behavior without delivering it.

The `contract_heavy` branch doubles `contract_fit` — the only branch producing a genuinely different score. However, `contract_fit` depends on `task_contract` being provided to `select_applicable_concepts`. The retriever passes `task_contract=task_contract` from its own parameter, but this is optional. None of the new tests pass a contract, so the 2x multiplier's effect on selection ranking remains unverified.

## semantic_balanced secondary admission: unreachable under current config

The secondary admission rule:

```python
if not select and strategy == 'semantic_balanced' and len(selected) == 1 and fam_limit >= 2:
    if adjusted >= secondary_threshold:
        select = True
```

This fires only for synthesis tasks (the sole `semantic_balanced` family). But `DEFAULT_FAMILY_SELECTIVITY['synthesis']['max_active']` is 1, so `fam_limit` is 1, and `fam_limit >= 2` is always false.

The documentation in `Virtual_SIA_Concept_Engine_Deep_Analysis_AR.md` Section 3.2 describes this rule as active: "allows synthesis tasks to benefit from two complementary concepts." This is aspirational — describing intent, not reachability. The eval ablation confirms: `synthesis_top2` (which manually overrides `max_active=2`) produces identical results because no second concept is competitive, but the implementation also structurally prevents it under default config.

## Retriever cap asymmetry across candidate families

After the multi-family loop collects concepts, the retriever caps with:

```python
primary_family_cfg = DEFAULT_FAMILY_SELECTIVITY.get(task_family, {})
family_max_active = primary_family_cfg.get('max_active', DEFAULT_GLOBAL_MAX_ACTIVE_CONCEPTS)
applicable_concepts = family_selected[:family_max_active]
```

If a task is classified as `procedure` (max_active=0), all concepts collected from secondary candidate families like `comparison` or `synthesis` are discarded — the slice is `[:0]`. The multi-family loop work is wasted. This may be intentional (procedure tasks should not receive concepts), but the design silently depends on the primary family always being the most restrictive gateway.

## Test coverage: what's missing

Not tested:
- Any test with a non-None `task_contract` (the `contract_heavy` 2x multiplier is never exercised)
- The `structural_only` high-threshold fallback (score >= 10 admission for procedure)
- The retriever's per-family cap behavior (especially the procedure-as-primary truncation)
- The fixed variable name in `_set_selectivity` (ablation runner integration)
- The new `family_selectivity_detail.py` report generator

The test `test_select_applicable_concepts_respects_max_active` verifies that selection count is `<= 1` but does not assert that exactly 1 concept *is* selected — it would pass even if zero concepts were selected (e.g., if the scoring formula regresses to reject everything).
