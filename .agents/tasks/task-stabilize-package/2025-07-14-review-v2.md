# Family-specific concept selectivity, test scaffold, and deep analysis documentation (v2)

This change adds per-family strategy-differentiated scoring to the concept engine (`contract_heavy`, `semantic_balanced`, `structural_only`), fixes a variable-name bug in the ablation runner, replaces the retriever's global concept cap with a max-across-candidate-families cap, introduces 13 pytest tests, a `pyproject.toml`, and three architecture/status documentation files. The v1 review identified 6 issues; all have been addressed: synthesis `max_active` raised to 2, the redundant `structural_only` scoring branch removed, a `contract_heavy` test added, the retriever cap fixed to use `max()` across families, a `_set_selectivity` test added, and the build backend corrected.

Watch for: Documentation-code drift on synthesis `max_active` (confirmed). Two strategy branches remain untested — `semantic_balanced` secondary admission and `structural_only` high-threshold fallback (confirmed). The `test_select_applicable_concepts_respects_max_active` test asserts `<=` rather than exact count, making it pass even under total scoring regression (confirmed).

**Verdict**: APPROVED

## High-level view

The scoring engine now genuinely differentiates: `contract_heavy` doubles `contract_fit` for comparison tasks, and the `structural_only` path is reduced to a comment explaining that its differentiation is the high-threshold fallback alone. With synthesis `max_active=2`, the `semantic_balanced` secondary admission rule is structurally reachable.

The retriever cap change from `family_selected[:global_cap]` to `family_selected[:max(candidate_families)]` resolves the original silent-drop concern. If `candidate_families = ["procedure", "comparison"]`, the cap is now `max(0, 1) = 1` rather than `0`, preserving comparison-sourced concepts.

`Virtual_SIA_Current_Regime_Status_AR.md` still states `synthesis: max_active=1` in its frozen-defaults table and rationale section, while the actual code now ships `max_active=2`. This drift will confuse any reader who uses the doc as the source of truth for current operating parameters.

The test suite covers the contract_heavy 2x multiplier, the `_set_selectivity` config mutation, contrastive group construction, task ingress, and eval runner structure. Two strategy-specific behaviors remain untested: the secondary admission rule and the structural_only high-threshold fallback.

## Documentation-code drift on synthesis max_active

The v1 review flagged that synthesis `max_active=1` made the secondary admission rule dead code. The fix correctly raised it to 2 in `config.py`. However, `Virtual_SIA_Current_Regime_Status_AR.md` was not updated:

```markdown
# Section 1.2 still shows:
'synthesis': {'max_active': 1, 'min_score': 7},

# Section 1.3 rationale table still says:
| Synthesis max_active | 1 | synthesis_top2 override shows no improvement |

# Section 3.1 still concludes:
- **Decision**: Keep synthesis at max_active=1 until the concept set grows
```

The FEAT-004 findings note that `synthesis_top2` now uses the default (since the default *is* 2), producing identical eval results. The doc should reflect that the ceiling for secondary concept admission is now open, even if the current concept set doesn't yet produce a competitive second candidate.

## Untested strategy branches

The **semantic_balanced secondary admission** (`apply.py` lines 106-109) fires when `strategy == 'semantic_balanced' and len(selected) == 1 and fam_limit >= 2 and adjusted >= fam_min_score`. With synthesis now at `max_active=2`, this is reachable. A test supplying two high-scoring synthesis-scoped concepts and asserting `len(selected) == 2` would exercise it.

The **structural_only high-threshold fallback** (`apply.py` lines 112-114) fires when `strategy == 'structural_only' and len(selected) == 0 and adjusted >= 10`. This is the only way a procedure task can receive a concept under the current config (`max_active=0, min_score=99`). A test supplying a procedure-scoped concept with artificially high semantic overlap would validate this escape hatch.

These are non-blocking because the branches are structurally simple (two-condition gates), but their absence means a refactor could silently break the only path through which procedure tasks receive concepts.

## Weak assertion in max_active test

```python
def test_select_applicable_concepts_respects_max_active():
    ...
    assert len(selected) <= DEFAULT_FAMILY_SELECTIVITY["comparison"]["max_active"]
```

If the scoring formula regresses to reject all concepts (e.g., `semantic_fit` calculation breaks), this test still passes. A tighter assertion:

```python
assert len(selected) == 1  # exactly one concept admitted, capped at max_active
```
