# Self-benchmarking eval loop and productive forgetting for memory lifecycle

This change adds two gated features to Virtual-GENESIS. Self-benchmarking (H8) turns anomaly signals from the eval system into new diagnostic test cases, discovers blind spots in the coverage matrix, and measures how well each test discriminates between conditions. Productive forgetting (H3) gives the memory store lifecycle management: decay scoring, archive/deprecate/delete operations, utility computation, and an automated forgetting policy that runs after each pipeline pass when enabled. Both features are behind boolean flags (`use_self_benchmarking`, `use_productive_forgetting`) that default to `False`, preserving backward compatibility. Documentation adds 10 Legitimate Thefts (5.45-5.54) sourced from FunSearch, Novelty Search, Ebbinghaus, MemOS and others, plus implementation memos and a regime status update.

**Watch for:** The pipeline passes `store.all()` to the retriever even with forgetting enabled, then filters inside the retriever — meaning archived/deprecated memories are loaded and traversed every call (confirmed). The decay function applies a linear staleness penalty with no floor on per-tick granularity, so calling `apply_decay` multiple times within the same tick compounds the deduction (confirmed). The forgetting policy's archive-vs-deprecate decision creates a race: a memory exactly at `threshold * 0.5` gets deprecated while one slightly above gets archived only if the archive quota hasn't been exhausted first (confirmed).

## High-level view

Self-benchmarking assembles four modules into a closed loop: `benchmark_generator` maps anomaly candidates to TaskCase objects via template prompts, `blind_spot_discovery` computes a coverage matrix over (family, perturbation, difficulty) tuples to find untested or suspiciously-easy regions, `diagnostic_value` uses the formula `4p(1-p)` to quantify how well each test splits conditions, and `run_self_benchmark_cycle` orchestrates the full cycle. The loop is purely combinatorial — no LLM calls, no external search — so it expands the eval surface only by the patterns the existing anomaly detector already notices.

Productive forgetting extends MemoryUnit with four new fields (decay_score, last_accessed, access_count, memory_status) and adds lifecycle operations to the store. Utility scoring combines retrieval frequency, recency, decay, and outcome quality with fixed weights — but the retrieval frequency signal is dead because nothing calls `record_access` during normal operation, reducing the scorer to three effective signals.

The pipeline wiring passes `store.all()` to `retrieve_memory` and re-filters with `active_only`. Decay is applied unconditionally on every pipeline call when the flag is on — there is no idempotency guard, so running two pipeline calls at the same tick doubles the penalty.

## Pipeline wiring and the double-filter path

The minimal pipeline calls `store.all()` and passes the full list to `retrieve_memory`, which then re-filters with `active_only=use_productive_forgetting`:

```python
memory_pack = retrieve_memory(
    task.task_family,
    task.normalized_text,
    store.all(),  # <-- all items including archived/deprecated
    ...
    active_only=use_productive_forgetting,
)
```

Inside the retriever:
```python
if active_only:
    store_items = [item for item in store_items if getattr(item, 'memory_status', 'active') == 'active']
```

Pass `store.get_active_memories()` when forgetting is enabled. The current path allocates a list of all items, filters it, then iterates the filtered list — three traversals where one would do.

## Decay mechanics and compounding risk

`apply_decay` computes `staleness = (self._tick - mem.last_accessed) * decay_rate` and subtracts it from `decay_score`. The store's `_tick` only advances on `store_memory` calls, so within a single pipeline run this is stable. But if the pipeline is called twice without any new memory being stored between calls (e.g., a task that doesn't produce an episode), the same tick value produces the same subtraction — the score decays by the same amount twice. With `decay_rate=0.05` and a staleness gap of 10 ticks, two calls produce `2 * 10 * 0.05 = 1.0` total deduction, zeroing out a memory that should only lose 0.5.

In the current flow, `store_memory` is always called before the end of the pipeline, so this isn't triggered on the happy path. It becomes a problem if the pipeline is extended with early-return paths or if `apply_decay` is called from multiple integration points.

## Forgetting policy archive-vs-deprecate semantics

The policy iterates sorted-by-utility-ascending and applies two rules:
1. If `utility < deprecation_threshold` (i.e., `threshold * 0.5`): deprecate.
2. Else if `utility < utility_threshold` and archive quota not exhausted: archive.

Deprecation is permanent (no pathway back to active), while archiving is presumably recoverable. The boundary at `0.5 * threshold` is a hard cliff — a memory at utility 0.149 gets deprecated while one at 0.151 gets archived. The policy doesn't log which threshold triggered the decision beyond the action name, making forensic analysis harder if memories disappear unexpectedly.

## Blind spot discovery and the perturbation axis sparsity

The coverage matrix keys on `(family, perturbation_type, difficulty_class)`. For regular pipeline runs, `stress_type` is not populated on the result dict's `task` field — it's only set by `benchmark_generator` for auto-generated cases. The extraction logic falls through to `diagnostic_purpose[0]` or defaults to `"none"`. Organic test results cluster into `(family, "none", difficulty)` tuples, and the discovery algorithm treats any auto-generated case with a different perturbation tag as "untested." This inflates the untested count in a way that doesn't reflect actual coverage gaps — the coverage ratio will always be low until the self-benchmark loop has run several iterations to populate those generated slots.

## Memory utility and the dead retrieval_frequency signal

The utility formula weights `retrieval_frequency` at 0.25, computed as `access_count / (access_count + 5)`. But `access_count` is only incremented by `store.record_access()`, which is never called by the retriever or the pipeline. The field stays at zero for all memories throughout normal operation, making this component a constant 0.0 contribution. Utility therefore reduces to recency (0.30), decay (0.25), and outcome quality (0.20) — effectively a three-signal scorer.

Wiring `record_access` into the retriever when a memory is selected would activate this signal.

## Diagnostic value and sample-size blindness

The formula `4 * p * (1 - p)` doesn't account for the number of conditions — a 1/1 split (n=2) and a 5/5 split (n=10) both score 1.0, despite the latter being far more statistically meaningful. A sample-size correction (e.g., adjusting toward 0.5 for small n) would prevent over-confidence in diagnostic value from tests run against only 2 conditions.

## Test coverage gaps

Not tested: the compounding decay scenario (multiple `apply_decay` calls at the same tick), the archive/deprecate boundary behavior when a memory sits exactly at `threshold * 0.5`, the interaction between self-benchmarking and productive forgetting running simultaneously, and what happens when the pipeline's forgetting threshold (>10 active) is crossed mid-run by the policy itself archiving memories.
