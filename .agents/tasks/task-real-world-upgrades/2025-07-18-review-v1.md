# Contract perturbations, SQLite persistence, and real LLM evaluation

Three features land together: contract-level perturbation operators that attack the verification criteria rather than prompt text, a full SQLite persistence layer mirroring the in-memory stores, and a real LLM evaluation runner comparing raw/concept-augmented/theory-augmented conditions via OpenRouter. The approach is conservative (new code paths gated behind `use_persistence=False`, eval runner only runs via `__main__`, existing tests untouched) and the test coverage is substantial (111 new tests).

**Watch for:** A live API key is hardcoded in `config.py` and committed to the repository (confirmed). The persistence layer opens and closes a new SQLite connection on every single method call, which will degrade under any real load (confirmed). The `run_single_condition` function builds a prompt it never sends, always routing through `generate_with_concepts` regardless of condition (confirmed).

## High-level view

The contract perturbation operators are cleanly designed: six operators that clone a TaskCase and modify `required_properties`/`forbidden_shortcuts` rather than `prompt_text`, plugged into new curriculum levels 6-7. The operators compose well because they follow the established `clone_case` + mutate + tag pattern. The curriculum now reaches 8 levels, and the lazy import in `build_curriculum_levels` sidesteps the circular dependency without polluting module scope.

The persistence package provides SQLite-backed implementations of the four core stores plus a checkpoint mechanism, all using stdlib `sqlite3` with WAL mode. The interface fidelity to the in-memory stores is high, and the Session constructor cleanly gates the switch. The cost is a connection-per-call pattern that precludes any concurrent or high-throughput use, and `initialize_database` being called in every store constructor means schema creation DDL runs repeatedly even when tables exist.

The LLM evaluation system adds cost tracking to `LLMAdapter`, a prompt-building module (`llm_reasoning.py`) that constructs condition-appropriate prompts, and a runner that evaluates 6 tasks across 3 conditions. All tests mock HTTP correctly. The critical gap is a committed API key and a redundant prompt path in the runner where the condition-specific prompt is constructed but `generate_with_concepts` is always called with the task text directly.

<summary>Issues (6)</summary>

1. **Hardcoded API key committed** — `config.py` contains a live `sk-or-v1-...` key. Rotate the key immediately and replace with env-var-only resolution or a placeholder.
2. **Connection-per-call in persistence** — Every store method opens and closes a connection. Use a persistent connection (or connection pool) per store instance.
3. **Redundant prompt construction in eval runner** — `run_single_condition` builds a condition-specific prompt via `build_raw_prompt`/`build_augmented_prompt` but then calls `generate_with_concepts` which builds its own prompt internally. The local `prompt` variable is used only for `prompt_length` in the result dict.
4. **Repeated schema DDL on every store instantiation** — `initialize_database` runs full `CREATE TABLE IF NOT EXISTS` DDL each time a store is constructed. Use a module-level flag or check schema_version before re-running.
5. **`_tick` state lost across persistence layer instances** — `SQLiteMemoryStore._tick` starts at 0 per instance; if multiple instances access the same database (e.g., across sessions), decay calculations based on tick differences become incorrect.
6. **LLM error responses pass through verification silently** — When `generate_with_concepts` returns `"[LLM_ERROR] ..."`, the string is fed directly to `verify_output` which pattern-matches it like any response. The eval results record this as `good_enough: False` but don't distinguish network failures from genuine verification failures.

## Hardcoded API key in config.py

Line 8 of `virtual_genesis/api/config.py`:

```python
OPENROUTER_API_KEY = os.environ.get("OPENROUTER_API_KEY", "")
```

This is a live OpenRouter key committed to version control. The key resolution chain in `LLMAdapter.__init__` prioritizes the constructor argument, then env var, then (if `use_config_fallback=True`) this constant. But `llm_reasoning.py` uses the config constant directly as a fallback in `generate_with_concepts`:

```python
resolved_key = api_key or os.environ.get("OPENROUTER_API_KEY") or OPENROUTER_API_KEY
```

The test `test_api_key_from_config_fallback` explicitly asserts the key starts with `"sk-or-"`, coupling test correctness to the presence of a real credential. The key should be rotated, the constant should become `""` or `None`, and tests should mock the env var rather than relying on a committed secret. (confirmed)

## Connection lifecycle in the persistence layer

Every public method on `SQLiteMemoryStore`, `SQLiteConceptRegistry`, `SQLiteTheoryRegistry`, and `SQLiteIdentityStore` calls `self._connect()` which creates a new `sqlite3.connect(self._db_path)` and sets `row_factory`. The connection is then closed in a `finally` block. For the memory store's `apply_decay`, this means one connection opens, fetches all active memory rows, iterates with individual UPDATE statements, commits, and closes.

This pattern is safe for single-threaded prototype use and avoids holding connections across long-lived objects. But it means WAL mode's concurrency benefits are never realized (no reader connection is held while a writer commits), and it makes the stores unsuitable for any request-serving context without refactoring. The `initialize_database` call in each constructor compounds this: constructing a `SQLiteMemoryStore`, a `SQLiteConceptRegistry`, and a `SQLiteTheoryRegistry` in `Session.__init__` runs the full DDL three times against the same database.

## `_tick` as instance state vs. persistent state

`SQLiteMemoryStore._tick` is an in-memory counter initialized to 0. The `store_memory` method increments it and stamps `unit.last_accessed = self._tick`. The `apply_decay` method computes staleness as `(self._tick - row["last_accessed"]) * decay_rate`.

If a new `SQLiteMemoryStore` instance is created pointing at an existing database (the cross-instance persistence test exercises exactly this), `_tick` resets to 0 while `last_accessed` values in the database may be large integers from the previous instance. The staleness calculation then produces negative values (clamped to 0 by `max(0.0, ...)`), causing decay to have no effect until the new instance's tick exceeds all stored `last_accessed` values. This isn't a crash, but it makes decay semantically incorrect across process restarts. (confirmed)

## Redundant prompt path in run_real_llm_eval

`run_single_condition` builds a condition-specific prompt:

```python
if condition == "A_raw":
    prompt = build_raw_prompt(task.prompt_text)
elif condition == "B_concept":
    prompt = build_augmented_prompt(task.prompt_text, concept_hints)
else:
    prompt = build_augmented_prompt(task.prompt_text, concept_hints, theory_hints)
```

Then immediately calls:

```python
response = generate_with_concepts(
    task.prompt_text,
    concept_hints=concept_hints if condition != "A_raw" else None,
    theory_hints=theory_hints if condition == "C_full" else None,
    api_key=resolved_key,
)
```

`generate_with_concepts` internally calls `build_augmented_prompt` again with the same arguments and sends *that* prompt to OpenRouter. The locally-built `prompt` variable is only used for `len(prompt)` in the result dict, making the reported `prompt_length` a lie for condition A_raw (it reports the raw prompt length, but what's actually sent is an augmented prompt with no hints, which has different formatting). (confirmed)

## Contract perturbation operator design

The six operators are well-structured. Each follows clone-mutate-tag without side effects on the input case. The `contract_flip` operator's conditional logic (`if len(new_case.forbidden_shortcuts) > 1`) means a case with exactly one forbidden shortcut won't get anything moved from shortcuts to properties. The first property still moves to shortcuts, but the single existing shortcut stays put. This is intentional (the `> 1` check prevents moving the item that was just appended), but the asymmetry means `contract_flip` doesn't always produce a true swap. For cases starting with one property and one shortcut: the property moves to shortcuts (now 2 shortcuts), then `len > 1` is true so the original shortcut moves to properties. For cases starting with one property and zero shortcuts: the property moves (now 1 shortcut), `len > 1` is false, so no shortcut moves to properties. The resulting case has 0 properties and 1 shortcut. (confirmed)

## Test coverage gaps

The persistence tests use `tempfile.mktemp` which creates a race condition (another process could create the file between name generation and use). `tempfile.NamedTemporaryFile(delete=False)` or `tmp_path` fixture would be safer, though in a test context this is low-risk.

Not tested: behavior of SQLite stores under concurrent access from multiple threads (the `SessionManager` is thread-safe but the stores it creates are not). Not tested: what happens when `generate_with_concepts` is called with `api_key=None` and no env var and the config constant is empty (currently it would call OpenRouter with an empty bearer token). Not tested: `run_real_llm_eval` end-to-end (appropriately avoided since it makes real API calls, but the interaction between `time.sleep` delays and error accumulation in a long run is untested).
