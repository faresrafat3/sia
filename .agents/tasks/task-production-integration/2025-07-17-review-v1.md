# Production integration: API server, full-flags runner, research paper, and merge strategy

This change adds four components to Virtual-GENESIS: a full integration test runner that exercises all five governance flags simultaneously across three task sets, a production REST API built on stdlib `http.server` with OpenRouter LLM integration, a formal Arabic research paper draft, and a merge strategy document for the PR chain. The code is pure-stdlib Python, and the tests bring the total count from 285 to 297.

Watch for: The API server has **no thread-safety on shared mutable state** (confirmed) — `session_manager` and `total_tasks_processed` are mutated from concurrent handler threads without locks. The `LLMAdapter` is **instantiated but never actually called** in the task flow (confirmed) — tasks go through `run_minimal_pipeline` which does its own simulated reasoning, making the OpenRouter integration dead code. Closed sessions can still accept task submissions because **session state is never checked before use** (confirmed).

## High-level view

The integration runner (`run_full_integration.py`, 321 lines) defines three condition functions — all-flags, concept-only, and combined-baseline — each following the established `run_condition` pattern from earlier eval runners. All five governance flags are correctly threaded through to `run_minimal_pipeline` in the all-flags condition. The runner reuses shared memory/concept/theory stores across tasks within a condition, building up state progressively.

The API (`virtual_genesis/api/`, 4 modules) provides a minimal REST surface: task submission, session lifecycle, health, and status. It routes HTTP via `BaseHTTPRequestHandler`, manages sessions in-memory with a `SessionManager`, and includes an `LLMAdapter` for OpenRouter that falls back to deterministic mock responses when no API key is present. The server is threaded (default `HTTPServer` with `serve_forever`), but the handler accesses shared state without synchronization.

The research paper (747 lines, Arabic) frames the project's results as a formal academic paper with 9 sections, citing the evaluation numbers from earlier PRs. The merge strategy document (216 lines, Arabic) specifies squash-merge ordering for PRs #2-#8 with a verification checklist.

1. **No thread-safety on server state** — `session_manager.sessions` dict and `total_tasks_processed` counter are mutated from concurrent handler threads. Add a `threading.Lock` around mutations, or switch to `ThreadingHTTPServer` with explicit locking.
2. **LLM adapter never called in task flow** — `self.server.llm_adapter` is constructed but `_handle_task` calls `run_minimal_pipeline` directly without involving the adapter. Either wire the adapter into the pipeline or remove the dead integration path.

4. **Unbounded session growth** — `end_session` sets state to "closed" but never removes the session from the dict. Long-running servers will accumulate dead sessions indefinitely. Add a removal or expiry mechanism.
5. **CORS preflight fails** — CORS headers advertise `OPTIONS` support but `do_OPTIONS` is not implemented, so browsers will get 501 on preflight requests. Implement `do_OPTIONS` returning 204 with the CORS headers.
6. **Dead `hashlib` import** — `llm_adapter.py` imports `hashlib` but never uses it. Remove the import.
7. **Integration test `test_all_flags_ge_concept_only` asserts a guarantee the system cannot make** — the test asserts that all-flags success rate >= concept-only on a 3-task sample. With such a small sample and deterministic pipeline, this passes trivially but would be fragile under any non-trivial change. Consider documenting why this invariant holds or making the assertion directional with a tolerance.

## Thread-safety gap in the API server

`HTTPServer.serve_forever()` processes requests sequentially by default, but the test setup and documentation suggest concurrent use. More critically, if anyone subclasses to `ThreadingHTTPServer` (or the Python runtime changes default behavior), every handler invocation shares `self.server.session_manager` and `self.server.total_tasks_processed` without synchronization.

In `_handle_task`:
```python
self.server.total_tasks_processed += 1
if session:
    session.task_count += 1
```

And in `_handle_session_start`:
```python
session = manager.create_session()
```

These are all unsynchronized mutations of shared dicts and counters. The `sessions` dict in `SessionManager` is modified by `create_session`, `end_session`, and read by `get_session` and `list_active` — all from handler methods that could overlap under threaded serving.

The fix is straightforward: add a `threading.Lock` to `SessionManager` and acquire it in each public method. For the counter, use `threading.Lock` or `itertools.count`.

## LLM adapter: constructed but never invoked

`SIAServer` stores `self.llm_adapter` (an `LLMAdapter` instance configured with the OpenRouter API key). However, `_handle_task` calls `run_minimal_pipeline` directly — the pipeline does its own simulated reasoning and never calls an external LLM. The adapter is dead weight in the current flow.

This means the advertised "OpenRouter integration" is structurally present (the adapter can make HTTP calls to OpenRouter) but functionally disconnected from the task submission endpoint. A user hitting `POST /task` will get pipeline simulation results, not LLM-generated responses, regardless of whether `OPENROUTER_API_KEY` is set.

If the intent is to eventually replace the pipeline's simulated reasoning with real LLM calls, the adapter needs to be passed into `run_minimal_pipeline` or injected at the reasoning step. Currently it's a dead code path.

## Closed session reuse

`_handle_task` retrieves sessions via:
```python
session = manager.get_session(session_id)
if session is None:
    self._send_json(404, {"error": "session_not_found"})
    return
```

But `get_session` returns any session regardless of state. A client can submit tasks to a closed session, mutating the memory store and concept registry that were already consolidated by `apply_forgetting_policy`. This violates the lifecycle invariant (create -> use -> consolidate -> close). The fix is a state check:

```python
if session.state != "active":
    self._send_json(409, {"error": "session_closed"})
    return
```

## CORS preflight gap

The response headers include:
```python
self.send_header("Access-Control-Allow-Methods", "GET, POST, OPTIONS")
```

But `SIAHandler` only implements `do_GET` and `do_POST`. When a browser sends a CORS preflight `OPTIONS` request, `BaseHTTPRequestHandler` will respond with 501 (Not Implemented). Any cross-origin browser client will fail on preflight. Implementing `do_OPTIONS` to return 204 with the CORS headers fixes this.

## Integration runner: condition functions and flag threading

The three condition functions in `run_full_integration.py` correctly differentiate their flag usage:

- `run_condition_all_flags`: passes all 5 governance flags + identity + concepts + economy
- `run_condition_concept_only`: only `use_concepts=True`, `use_memory=True`, `forced_tier="tier_1"`, economy disabled
- `run_condition_combined_baseline`: `use_concepts=True`, `use_economy=True`, no governance flags

All three share the same structural pattern — warmup phase, per-task loop with concept/theory refresh, metric aggregation. The code is repetitive (each function is ~50 lines of nearly identical loop logic), but this is defensible for clarity: each condition is self-contained and independently testable.

The `interaction_effects` calculation in `integration_summary.py` defines synergy as `(all_flags_rate - concept_only_rate) - (combined_rate - concept_only_rate)`, which simplifies to `all_flags_rate - combined_rate`. This measures whether adding governance flags on top of the combined baseline improves success, which is a valid interaction metric but isn't measuring multi-mechanism synergy in the strict sense (it doesn't decompose individual mechanism contributions).

## Test coverage shape

The API tests (`test_api.py`, 8 tests) spin up a real server with `port=0` and make HTTP requests via `urllib`. They cover the happy path for all endpoints and verify the LLM adapter's mock mode. The session lifecycle test (`test_task_with_session`) exercises create -> submit task -> end, confirming `task_count` increments.

Not tested: error paths (malformed JSON, missing Content-Length, oversized payloads), concurrent request handling, behavior when submitting to a closed session, and the actual OpenRouter code path (which requires a live API key).

The integration tests (`test_full_integration.py`, 4 tests) verify that all-flags pipeline produces correct result structure, that all governance keys appear in output, and that the runner functions return valid payloads. The `test_all_flags_ge_concept_only` assertion is structurally weak — with deterministic simulated reasoning and 3 tasks, both conditions likely achieve identical success rates, making the assertion vacuously true rather than testing a real property.

## Unbounded session accumulation

`SessionManager.end_session` transitions state to "closed" but the session object remains in `self.sessions` forever. Over time, a long-running server accumulates closed sessions with their full memory stores, concept registries, and theory registries in memory. There is no TTL, garbage collection, or explicit removal.

For a prototype this is acceptable if documented, but the merge strategy document positions this as production-ready, which creates a mismatch between documentation claims and implementation reality.
